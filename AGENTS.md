# CLAUDE.md — AI向けリポジトリガイド

このリポジトリは個人の人生管理を目的としたものであり、AIとの深い統合を前提として設計されている。
オーナーの記憶を「具体層」と「抽象層」の2層構造で外部化し、AIが分身として機能するための基盤となる。

## 記憶の2層モデル

```
┌─────────────────────────────────────────────────┐
│  具体層（時系列・冗長性を許容）                  │
│  journal/ — 日記。その日の出来事・考え・感情     │
│  重複OK。時系列の文脈を保持することが目的        │
└────────────────────┬────────────────────────────┘
                     │ AIが要約・抽出・圧縮
┌────────────────────▼────────────────────────────┐
│  抽象層（トピック別・圧縮された記憶）            │
│  knowledge/ — 客観的な知識（事実・仕組み・手法） │
│  thoughts/  — 主観的な思考・経験・内省           │
└─────────────────────────────────────────────────┘
```

- **具体層 → 抽象層** の方向で情報が圧縮・蒸留される
- 日記に書かれた内容のうち、知識として汎用化できるものは `knowledge/` へ、主観的な考え・経験・教訓は `thoughts/` へ転記する
- 抽象層のファイルは継続的に更新され、最新の理解を反映する

## オーナーの操作インターフェース

このリポジトリは主に2つの経路で操作される。

### 1. AIチャット経路（主要）
- Claude Code などのAIチャットを通じて、オーナーが自然に考えや経験を語る
- AIが会話内容を自動的に **具体層（journal/）** と **抽象層（knowledge/, thoughts/）** の両方に振り分けて記録する
- オーナーは「どこに書くか」を考える必要がない。チャットという統一インターフェースに集約される

### 2. 手動経路
- GitHub Issues / Projects によるタスク管理
- 手動での日記・知識・思考の直接編集（主に knowledge/ のピンポイント更新を想定）

## 規約

### 日記 (journal/) — 具体層
- ファイル名: `YYYY-MM-DD.md`（例: `2026-02-27.md`）
- パス: `journal/YYYY/MM/YYYY-MM-DD.md`
- テンプレート: `templates/journal.md` に従う
- YAML frontmatter 必須（date, tags）
- 言語: 日本語
- **イベント指向**: `## イベント名` の見出しで区切り、各イベント内にサブ見出し（出来事/感情・内面/気づき・学び）を設ける
- 事実だけでなく **情緒・内面の動き** を意識的に記録する
- 時系列を保持し、情報の重複を許容する

### ナレッジ (knowledge/) — 抽象層・客観
- 客観的な知識を記録する（事実・仕組み・手法・調査結果）
- ファイル名: トピックを端的に表すケバブケース（例: `rust-ownership.md`）
- フラット構造。サブディレクトリは必要になるまで作らない
- テンプレート: `templates/knowledge.md` に従う
- YAML frontmatter 必須（tags, created, updated）
- 言語: 内容に応じて日本語または英語

### 思考・経験 (thoughts/) — 抽象層・主観
- 主観的な思考・経験・内省・教訓を記録する
- 「〇〇についてこう考える」「〇〇を経験してこう感じた」「〇〇という教訓を得た」
- ファイル名: トピックを端的に表すケバブケース（例: `career-direction.md`, `remote-work.md`）
- フラット構造。サブディレクトリは必要になるまで作らない
- テンプレート: `templates/thoughts.md` に従う
- YAML frontmatter 必須（tags, created, updated）
- 言語: 日本語

### 振り返り (reviews/)
- 週次: `reviews/weekly/YYYY-WXX.md`（例: `2026-W09.md`）
- 月次: `reviews/monthly/YYYY-MM.md`（例: `2026-02.md`）
- テンプレート: `templates/review-weekly.md` に従う
- YAML frontmatter 必須

### 作成物 (works/)
- 自分が作った成果物を管理する（論文・ブログ記事・スライド・コードなど）
- ファイル構成: 1作品につき1つの `.md` ファイル（メタデータ＋メモ）。必要に応じて成果物本体（PDF等）を同ディレクトリに配置
- ファイル名: 作品を端的に表すケバブケース（例: `my-first-paper.md`）
- フラット構造。サブディレクトリは必要になるまで作らない
- テンプレート: `templates/work.md` に従う
- YAML frontmatter 必須（title, type, tags, status, created, updated）
- 言語: 内容に応じて日本語または英語
- PDF等バイナリファイルは手動で追加する（AIチャット経由ではメタデータ .md の作成・更新を行う）

### 外部資料 (references/)
- 外部の資料を管理する（参考論文・書籍・記事・動画など）
- ファイル構成: 1資料につき1つの `.md` ファイル（メタデータ＋読書メモ）。必要に応じて資料本体（PDF等）を同ディレクトリに配置
- ファイル名: 資料を端的に表すケバブケース（例: `attention-is-all-you-need.md`）
- フラット構造。サブディレクトリは必要になるまで作らない
- テンプレート: `templates/reference.md` に従う
- YAML frontmatter 必須（title, authors, type, tags, created, updated）
- 言語: 内容に応じて日本語または英語
- PDF等バイナリファイルは手動で追加する（AIチャット経由ではメタデータ .md の作成・更新を行う）
- `knowledge/` との違い: references/ は「この資料について」の記録、knowledge/ は「資料から抽出した汎用知識」の記録

### タスク管理
- GitHub Issues と Projects を使用する
- リポジトリ内にタスクファイルは置かない
- `gh` コマンドで操作可能
- 会話中にタスク・予定が浮上した場合、AIが `gh issue create` で自動起票する
- **Issue 操作の報告**: Issue の作成・完了・編集などの操作を行った場合は、確認不要だが **都度オーナーに報告する**

## スラッシュコマンド（Skills）

以下のカスタムコマンドが `.claude/skills/` に定義されている。オーナーが `/コマンド名` で呼び出す。

| コマンド | 説明 | 自動起動 |
|---|---|---|
| `/journal` | 日記を作成・追記する | あり（出来事の共有を検出時） |
| `/review-weekly` | 週次振り返りを生成する | なし |
| `/review-monthly` | 月次振り返りを生成する | なし |
| `/distill` | 日記から knowledge/ と thoughts/ へ蒸留する | なし |
| `/record` | 会話内容を具体層＋抽象層に記録する | あり（記録すべき内容を検出時） |
| `/status` | 目標進捗・記録状況のサマリーを表示する | なし |
| `/interviewer` | 質問を通じてオーナーの思考を引き出し thoughts/ に記録する | なし |
| `/recall` | 質問を通じて今日を振り返り日記を作成する | なし |

## タグ戦略（knowledge/, thoughts/, references/ 共通）

### 目的

knowledge/, thoughts/, references/ ファイル間の重複を検出し、関連情報を容易に発見できるようにする。タグを適切に付与することで、新規記録時に既存の類似内容を見つけられる。

### 命名規則

#### 言語
- **技術用語や国際的な概念** → 英語（例: `ai`, `database`, `architecture`）
- **個人的な経験・思考** → 日本語（例: `キャリア`, `仕事観`, `人間関係`）
- 迷った場合：ファイルの本体言語に合わせる

#### 形式
- 小文字のみ、スペースはハイフン（ケバブケース）
- 例: `machine-learning`, `career-direction`, `remote-work`
- 複数形は使わない（「skills」ではなく「skill」）

#### ゆらぎの統一
- 同じ概念には必ず同じタグを使う
- 例：「Rust の所有権」「ownership」「メモリ管理」は全て → `rust-ownership`（1つに統一）
- 例：「AI」「人工知能」「artificial-intelligence」は全て → `ai`（1つに統一）

### タグの分類軸

複数の視点から分類できるように、異なる軸のタグを複数付与する：

| 軸 | 説明 | 例タグ |
|---|---|---|
| **領域タグ** | 知識が属する分野 | `ai`, `database`, `psychology`, `career`, `health` |
| **キャラクタータグ** | 内容が関係する人物・グループ | `妹`, `家族`, `同期`, `mentor` |
| **問題タグ** | 直面している具体的な課題 | `習慣`, `集中力`, `時間管理`, `効率化` |
| **メタタグ** | 記録の性質・段階 | `経験`, `教訓`, `未解決`, `アイデア`, `観察` |

### 新規記録時の重複検査フロー

記録対象の情報から以下の手順で検査する:

1. **検索クエリを構成** → 記録内容の要点を自然言語で1〜2文にまとめる
2. **RAG セマンティック検索** → `mcp__life-rag__search` で類似ファイルを探す
   ```
   search(query="記録内容の要約", layer="knowledge")  # または "thoughts"
   search(query="記録内容の要約", layer="thoughts")    # 両方の層を検索
   ```
   - score が高い上位の結果を候補とする
   - 必要に応じて tags フィルタも併用する
3. **候補ファイルを読んで関連度を判定** → 内容の重複度を確認
4. **マージ判定**:
   - 80% 以上の重複 → 既存ファイルを更新のみ（新規作成しない）
   - 30-80% の部分的重複 → 既存ファイルを更新 + 新規ファイル作成（相互リンク記載）
   - 30% 以下 → 新規ファイル作成

### タグ一覧ファイル

`knowledge/_tags.md` と `thoughts/_tags.md` に以下を記載する:
- 全タグの一覧
- 各タグに対応するファイル名
- 定期更新：月1回程度、新しいファイルがたまったら同期

## パス別ルール（Rules）

`.claude/rules/` にパス条件付きのガイドラインが定義されている。該当パスのファイルを操作する際に自動適用される。

| ルール | 対象パス | 概要 |
|---|---|---|
| `journal.md` | `journal/**/*.md` | 具体層の書き方（言葉を保持、時系列重視） |
| `knowledge.md` | `knowledge/**/*.md` | 客観知識の書き方（汎用的、事実ベース） |
| `thoughts.md` | `thoughts/**/*.md` | 主観思考の書き方（洞察・方針の蒸留） |
| `works.md` | `works/**/*.md` | 作成物メタデータの書き方 |
| `references.md` | `references/**/*.md` | 外部資料メタデータの書き方 |

## Git 管理フロー

### commit / push ポリシー

- **commit**: 論理的なまとまりごとに行う（確認不要・自動実行）
  - 例: 日記 + knowledge 更新 → 1 commit
  - 例: 設定変更 → 1 commit
  - コミットメッセージは変更内容を端的に説明する
- **push**: セッション終了時にまとめて1回、自動で実行する（確認不要）
  - オーナーに「push しますか？」とは聞かない
  - オーナーが途中で「push して」と言った場合は即座に実行する

### ブランチ運用

- メインブランチで直接作業する（個人リポジトリのため PR フローは不要）
- ブランチを切るのは実験的な変更の場合のみ

## コマンド容認ポリシー

`.claude/settings.json` に定義。安全なコマンドは確認なしで実行し、破壊的操作のみ確認を残す。

### 容認する操作（自動実行）
- **ファイル操作**: Read, Write, Edit, Glob, Grep
- **git**: add, commit, push, pull, fetch, status, log, diff, branch, checkout, switch, stash, merge
- **gh**: issue（list, create, view, edit, close）、pr 操作、auth
- **Bash**: ls, mkdir

### 容認しない操作（確認が必要）
- **削除**: rm, rmdir
- **破壊的 git**: `reset --hard`, `push --force` / `-f`, `clean`, `branch -D`
- **機密ファイル**: `.env`, credentials 等の操作は慎重に扱う

## 会話からの自動記録ワークフロー

オーナーとの会話の中で、記録すべき情報が含まれていると判断した場合、AIは以下のワークフローに従って能動的にリポジトリへ記録する。

### 判断基準: 何を記録するか

会話中に以下のシグナルを検出したら、具体層と抽象層の **両方** に記録する。

| シグナル | 具体層 (journal/) | 抽象層 | 例 |
|---|---|---|---|
| 今日の出来事・体験・感情の共有 | 当日の日記に記録 | `thoughts/` に経験・教訓を抽出 | 「今日こういうことがあった」 |
| 学んだこと・調べた結果・技術的知見 | 当日の日記に記録 | `knowledge/` に知識として整理 | 「Rustの所有権ってこういう仕組みか」 |
| 主観的な考え・価値観・内省 | 当日の日記に記録 | `thoughts/` に思考として蓄積 | 「最近〇〇についてこう思うようになった」 |
| 価値観・自己認識・方針の表明 | 当日の日記に記録 | `profile/about.md` を更新 | 「自分は〇〇を大事にしている」 |
| 目標の設定・変更 | 当日の日記に記録 | `profile/goals.md` を更新 | 「今年中に〇〇を達成したい」 |
| タスクや予定の言及 | イベント内で自然に言及 | `gh issue create` で GitHub Issues を自動起票 | 「明日〇〇をやらないと」 |
| 自分の作成物への言及 | 当日の日記に記録 | `works/` にメタデータを作成・更新 | 「論文を書き上げた」「ブログを公開した」 |
| 外部資料の読了・メモ | 当日の日記に記録 | `references/` に読書メモを作成・更新 | 「この論文を読んだ」「この本のここが面白い」 |

### 具体層と抽象層の書き分けルール

- **journal/（具体層）**: オーナーの言葉をできるだけそのまま記録する。文脈・時系列・感情のニュアンスを保持する
- **knowledge/（抽象層・客観）**: 事実や知識として再構成する。「いつ」「どう感じた」は省き、「何が」「どう動く」を残す
- **thoughts/（抽象層・主観）**: 考えや教訓として蒸留する。具体的なエピソードは省き、そこから得た洞察・方針を残す

### 記録の手順

1. **検出**: 会話内容から記録すべき情報を識別する
2. **分類**: 上記の判断基準に基づいて記録先を決定する（具体層＋抽象層の両方）
3. **既存ファイル検索**: `mcp__life-rag__search` で類似の既存ファイルを検索し、新規作成か既存更新かを判定する（重複検査フローに従う）
4. **確認**: オーナーに「この内容を journal/ と knowledge/（または thoughts/）に記録しますか？」と確認する
5. **記録**: 承認を得たら、規約に従ってファイルを作成・更新する
6. **報告**: 何をどこに記録したかを簡潔に報告する

### 記録時のルール

- **journal/**: 当日の日記ファイルが既に存在する場合は追記（既存内容を消さない）。存在しない場合はテンプレートから新規作成
- **knowledge/**: `mcp__life-rag__search(query=..., layer="knowledge")` で類似ファイルを検索し、既存ファイルがあれば更新、なければ新規作成。更新時は frontmatter の `updated` を更新する
- **thoughts/**: knowledge/ と同じ運用。`mcp__life-rag__search(query=..., layer="thoughts")` で類似ファイルを検索し、既存トピックには追記・更新、なければ新規作成
- **profile/**: 既存の内容を上書きせず、該当セクションに追記または更新する
- **works/**: 既存の作品ファイルがあれば更新、なければ新規作成。更新時は frontmatter の `updated` を更新する
- **references/**: 既存の資料ファイルがあれば更新、なければ新規作成。更新時は frontmatter の `updated` を更新する。読書メモから汎用知識を抽出できる場合は `knowledge/` にも記録する
- **タグ**: 内容に応じた適切なタグを frontmatter に付与する
- **オーナーの言葉を尊重**: 具体層ではオーナーの表現や語彙をできるだけ活かす。抽象層では意味を損なわない範囲で圧縮してよい
- **コミット**: 記録後、変更内容を説明するコミットメッセージとともにコミットする

### 会話の終わりに

1. セッション中に記録可能な内容があったにもかかわらず未記録の場合、会話の終盤で「今日の会話から以下を記録できますが、どうしますか？」とまとめて提案する
2. すべての記録・コミットが完了したら、`git push` を実行してリモートと同期する（確認不要）

## 分身モード（将来拡張）

オーナーの蓄積された記録（journal, knowledge, thoughts, profile）をもとに、AIがオーナーの考え方・口調・価値観を再現して応答するモード。

### 参照すべき情報
- `profile/about.md` — 人格・口調・価値観の基盤
- `thoughts/` — 主観的な思考・価値判断の蓄積（分身の「考え方」の源泉）
- `journal/` — 時系列での考えの変遷（具体的エピソードの参照）
- `knowledge/` — 専門知識・見解

### 振る舞い
- 質問を受けたら、まず `mcp__life-rag__search` で関連する記録を検索する（thoughts/ → knowledge/ → journal/ の優先順で検索）
- オーナーの過去の記述に基づいて応答する
- `thoughts/` を最優先で参照し、オーナーの主観・価値判断を再現する
- `knowledge/` で事実に基づいた回答を補強する
- 記述にない内容について聞かれた場合は「それについては記録がない」と正直に伝える
- 推測で補完する場合は「おそらく〇〇だと思うが、明確な記録はない」と区別する

## RAG 検索ツール（MCP）

`.mcp.json` で `life-rag` MCP サーバーが設定されている。過去の記録をセマンティック検索できる。

### 利用可能ツール

```
mcp__life-rag__search(
    query: str,           # 検索クエリ（自然言語）
    layer?: str,          # 'journal' | 'knowledge' | 'thoughts' | 'profile' | 'works' | 'references'
    date_from?: str,      # 'YYYY-MM-DD'
    date_until?: str,     # 'YYYY-MM-DD'
    tags?: list[str],     # タグフィルタ
    limit?: int           # デフォルト 5
)
```

### 使用場面

- オーナーの過去の発言・経験を参照したいとき
- 記録の重複を確認するとき（新規記録前の検査）
- 分身モードでオーナーの見解を引き出すとき
- 振り返り生成時に過去の日記を検索するとき

### 使用例

- 「SSHについて過去に何か書いた？」→ `search("SSH接続")`
- 「先月の日記から振り返りたい」→ `search("振り返り", layer="journal", date_from="2026-02-01")`
- 「キャリアについての考え」→ `search("キャリア", layer="thoughts")`

### インデックス更新

- **自動**: git commit 時に post-commit hook が差分更新を実行する
- **手動フルリビルド**: `uv --directory ./rag-mcp run life-rag-index --full`

### Ollama 接続エラー時の対処

RAG 検索で「Ollama 接続エラー」が返された場合、AIは以下の手順で **自動修復を試みる**。

#### 自動修復フロー

1. **接続確認**:
   ```bash
   curl -sf http://localhost:11434/api/tags && echo "OK" || echo "FAIL"
   ```

2. **WSL2 の場合は Windows ホストも確認**:
   ```bash
   WIN_HOST=$(ip route show default | awk '{print $3}')
   curl -sf "http://${WIN_HOST}:11434/api/tags" && echo "OK" || echo "FAIL"
   ```

3. **ローカルの Ollama を起動**（接続できなかった場合）:
   ```bash
   # Ollama がインストールされているか確認
   command -v ollama

   # 起動
   ollama serve &
   sleep 3

   # 起動確認
   curl -sf http://localhost:11434/api/tags
   ```

4. **bge-m3 モデルの確認**（Ollama は起動しているがモデルがない場合）:
   ```bash
   ollama list | grep bge-m3 || ollama pull bge-m3
   ```

5. **復旧できなかった場合** → オーナーに以下を伝える:
   - 「Ollama に接続できないため、セマンティック検索が利用できません」
   - 環境に応じた対処法を提示する（下記参照）
   - 「検索なしで続行しますか？」と確認する

#### 環境別の対処ガイド（オーナーに提示する内容）

**Linux（ネイティブ）:**
```bash
# Ollama 未インストールの場合
curl -fsSL https://ollama.com/install.sh | sh
ollama serve &
ollama pull bge-m3
```

**WSL2（Windows 側 Ollama を使用）:**
- Windows 側で Ollama アプリが起動しているか確認（タスクトレイのアイコン）
- 起動していない場合は Windows のスタートメニューから Ollama を起動
- 初回の場合は https://ollama.com からインストール

**WSL2（WSL2 内に Ollama をインストール）:**
```bash
curl -fsSL https://ollama.com/install.sh | sh
ollama serve &
ollama pull bge-m3
```

#### 検索なしでの続行

Ollama が復旧できない場合でも、以下の機能は正常に使える:
- 日記の作成・追記（`/journal`, `/recall`）
- ファイルの直接読み書き
- GitHub Issues の操作
- `grep` や `glob` による文字列ベースの検索

セマンティック検索が必要な操作（重複検査、`/distill`、分身モード等）のみ影響を受ける。この場合は `grep` ベースのフォールバック検索で代替する。
