# /journal — 日記の作成・追記

オーナーの会話内容を当日の日記に記録する。

## 手順

1. 今日の日付を確認する
2. `journal/YYYY/MM/YYYY-MM-DD.md` が存在するか確認する
   - **存在しない場合**: `templates/journal.md` をもとに新規作成する。frontmatter の date を今日の日付に設定する
   - **存在する場合**: 既存内容を保持したまま追記する
3. `$ARGUMENTS` またはこれまでの会話内容から記録すべき内容を抽出する
4. イベント指向で構成する（後述の書き方ルールに従う）
5. frontmatter の tags を内容に応じて更新する
6. 会話の中で「明日やること」「〇〇しないと」「来週は〇〇」のようなタスク・予定が浮上した場合、`gh issue create` で GitHub Issue に自動起票する
7. 変更をコミットする

## 書き方ルール

### イベント指向の構造

日記は **イベント（出来事）ごとに `##` 見出し** を立てて記述する。リスト形式ではなく、物語のように各イベントを描写する。

各イベントには以下のサブ見出しを設ける:

- `### 出来事` — 何があったか
- `### 感情・内面` — そのとき何を感じたか、どんな気持ちだったか
- `### 気づき・学び` — そこから何に気づいたか、何を学んだか

すべてのサブ見出しを埋める必要はない。自然に書けるものだけでよい。

### 情緒・内面へのフォーカス

事実の記録だけで終わらせず、**感情・内面の動きを意識的に記録する**。

- オーナーが明示的に感情を語っていなくても、文脈から読み取れる感情があれば確認して記録する
- 「楽しかった」「不安だった」のような単語だけでなく、**なぜそう感じたか**まで掘り下げる
- 体調・気分・エネルギーレベルなど身体感覚も情緒の一部として扱う

### 基本ルール

- 具体層として、文脈・時系列・感情のニュアンスを保持する
- オーナーの語彙や表現を尊重する
- 重複を恐れない（時系列の記録として価値がある）

## タスクの自動起票

会話の中で以下のようなシグナルを検出したら、`gh issue create` で Issue を作成する:

- 「明日〇〇やらないと」「来週は〇〇」「〇〇しなきゃ」
- 具体的なタスクや予定への言及

起票後、日記のイベント内で自然に言及する（独立セクションは不要）。

## GitHub Issue 操作の報告

Issue の作成・完了・編集など、あらゆる Issue 操作を行った場合は **都度オーナーに報告する**。確認は不要だが、何をしたかは必ず伝える。

例:
- 「Issue #XX『〇〇』を作成しました」
- 「Issue #XX をクローズしました」

## 抽象層への蒸留

日記を書いた後、以下に該当する内容があれば抽象層への記録も提案する:
- 客観的な知識・学び → `knowledge/` へ
- 主観的な考え・教訓 → `thoughts/` へ
- 価値観・自己認識 → `profile/about.md` へ
- 目標の設定・変更 → `profile/goals.md` へ

蒸留先を決定する際は、`mcp__life-rag__search(query="蒸留内容の要約", layer="knowledge")` および `layer="thoughts"` で既存ファイルを検索し、新規作成か既存更新かを判定する。

必ず抽象層に記録したことをオーナーに報告する。
